# Copilot Metrics Dashboard

An automated GitHub Copilot effectiveness dashboard that goes beyond usage stats. Collects Copilot adoption metrics alongside PR health, issue lifecycle, and developer velocity data to help you answer the real question: **Is AI actually making your team more effective?**

> Measuring Copilot adoption without measuring developer outcomes is like counting gym memberships without checking if anyone got healthier.

## What It Measures

### Copilot Usage (from the Copilot Metrics API)
- **Acceptance rate** trends (daily/weekly)
- **Active vs. engaged users** over time
- **Suggestions and acceptances** by language and editor
- **Chat activity** (turns, insertions, copies)
- **Seat utilization** (active, inactive, never used)

### Pull Request Health (from the REST API)
- **PR lifespan** - median and P90 time from open to merge
- **Time to first review** - how long PRs wait for review
- **Review cycles** - rounds of review before merge
- **PR size** - additions + deletions (smaller = healthier)
- **Merge rate** - PRs merged vs. closed without merge
- **Weekly throughput** - PRs merged per week

### Issue Lifecycle (from the REST API)
- **Issue lifespan** - median and P90 time from open to close
- **Time to first response** - how quickly issues get attention
- **Stale issue count** - issues with no activity in 30+ days
- **Backlog trend** - is the issue backlog growing or shrinking?
- **Weekly throughput** - issues opened vs. closed per week

### Alerting
- Unused Copilot seats (wasting money)
- Acceptance rate dropping week-over-week
- Low active user percentage
- PR lifespan exceeding thresholds
- Slow time to first review
- Issue backlog growing for consecutive weeks
- Stale issues accumulating

## Architecture

```
GitHub Actions (nightly cron)
  ├── collect_copilot_metrics.py  → data/copilot/
  ├── collect_pr_metrics.py       → data/pulls/
  ├── collect_issue_metrics.py    → data/issues/
  └── generate_site.py            → site/data/dashboard.json
                                    site/data/alerts.json

GitHub Pages (static site)
  └── site/
      ├── index.html              ← Dashboard UI
      ├── css/dashboard.css
      ├── js/dashboard.js         ← Chart.js visualizations
      └── data/dashboard.json     ← Generated by workflow
```

## Quick Start

### 1. Fork or clone this repo

```bash
gh repo fork jmassardo/copilot-metrics-dashboard --clone
```

### 2. Configure secrets and variables

Go to **Settings > Secrets and variables > Actions** in your repo.

**Secrets:**

| Secret | Description |
|--------|-------------|
| `METRICS_GITHUB_TOKEN` | PAT or GitHub App token (see permissions below) |

**Variables:**

| Variable | Required | Description |
|----------|----------|-------------|
| `GITHUB_ORG` | Yes | Organization name to collect metrics for |
| `GITHUB_REPOS` | No | Comma-separated repos to track (auto-discovers if empty) |
| `COLLECTION_DAYS` | No | Days of history per run (default: 28) |
| `DASHBOARD_DAYS` | No | Days of data on dashboard (default: 90) |

### 3. Required token permissions

If using a **Personal Access Token (classic)**, you need:
- `manage_billing:copilot` - Copilot metrics and seat data
- `repo` - PR and issue data (for private repos)
- `read:org` - Organization repo discovery

If using a **Fine-grained PAT**, you need:
- Organization permissions: `Copilot Business` (read), `Members` (read)
- Repository permissions: `Issues` (read), `Pull requests` (read), `Metadata` (read)

If using a **GitHub App**, the app needs:
- Organization permissions: `Copilot Business` (read), `Members` (read)
- Repository permissions: `Issues` (read), `Pull requests` (read), `Metadata` (read)

### 4. Enable GitHub Pages

Go to **Settings > Pages** and set:
- Source: **GitHub Actions**

### 5. Run the workflow

Trigger manually via **Actions > Collect Metrics & Deploy Dashboard > Run workflow**, or wait for the nightly cron.

## Alert Thresholds

All thresholds are configurable via environment variables:

| Variable | Default | Description |
|----------|---------|-------------|
| `ALERT_SEAT_INACTIVE_DAYS` | 30 | Days before a seat is flagged inactive |
| `ALERT_ACCEPTANCE_RATE_DROP` | 10.0 | Percentage point drop to trigger alert |
| `ALERT_PR_LIFESPAN_HOURS` | 48 | Median PR lifespan threshold |
| `ALERT_TIME_TO_FIRST_REVIEW_HOURS` | 24 | Median TTFR threshold |
| `ALERT_ISSUE_BACKLOG_GROWING_WEEKS` | 3 | Consecutive weeks of backlog growth |
| `ALERT_MIN_ACTIVE_USER_PCT` | 50.0 | Minimum active seat percentage |
| `ALERT_NEW_SEAT_INACTIVE_DAYS` | 14 | Days before new seat flagged unused |

## Local Development

```bash
# Install dependencies
pip install -r requirements.txt

# Set environment variables
export GITHUB_TOKEN="ghp_your_token_here"
export GITHUB_ORG="your-org"
export GITHUB_REPOS="repo1,repo2"  # Optional

# Run collectors
python scripts/collect_copilot_metrics.py
python scripts/collect_pr_metrics.py
python scripts/collect_issue_metrics.py

# Generate dashboard
python scripts/generate_site.py

# Preview the dashboard
cd site && python -m http.server 8080
```
